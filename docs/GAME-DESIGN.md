# 玩法与案件设计 · GAME-DESIGN

> 一句话：**对话是软的（LLM 实时生成），判定是硬的（作者数据决定）。**

## 1. 核心循环

```
案件简报  ──►  审讯嫌疑人  ──►  找破绽 / 出示证据  ──►  指认  ──►  结案复盘
                  ▲   │ (LLM 实时对话)        │ (作者数据判定)
                  └───┘ 可随时切换嫌疑人、反复盘问
```

一局只有一桩案子、三名嫌疑人（1 真凶 + 2 无辜），玩家的目标是：**先靠对话锁定可疑的人，再用证据把他的谎言一条条钉死。**

## 2. 一桩案子由什么组成

数据模型见 [`src/data/types.ts`](../src/data/types.ts)，一桩案子（`CaseData`）= ：

| 字段 | 作用 |
| --- | --- |
| `clues` | 现场线索，简报里展示，给玩家推理用 |
| `evidence` | **可出示的证物**（纽扣 / 通话记录 / 监控…），审讯中甩给嫌疑人 |
| `suspects` | 嫌疑人：公开人设 + 隐藏 `systemPrompt`（实时）+ `offline` 剧本（离线） |
| `culpritId` | 真凶是谁 |
| `lieBreaks` | **「证据 → 谎言」映射**：把某证据出示给某嫌疑人，会击破某条谎言 |
| `requiredBreaks` | 定罪所需的最少铁证数（本案 = 2） |
| `solution` | 结案时揭晓的完整真相 |

## 3. 胜负判定：为什么不交给 LLM

`GameState.accuse()`（[`src/game/GameState.ts`](../src/game/GameState.ts)）的逻辑是纯函数：

```ts
correct = (accused === culprit)
if (correct && brokenLies ≥ requiredBreaks) → 定罪成功 convicted
else if (correct)                           → 证据不足 insufficient（真凶脱罪）
else                                        → 误判 wrong（冤枉好人）
```

而「某条谎言是否被击破」，由 `resolvePresent()` 查 `lieBreaks` 表决定，**完全不问 LLM**。

这带来三个好处：

1. **公平可解**：换任何模型、甚至离线，案子都有稳定解；
2. **防劝供**：嫌疑人在对话里再怎么「松口」也不等于定罪，定罪只认证据链；
3. **反作弊**：玩家无法靠 prompt 注入让游戏直接判他赢——判定不在模型手里。

> 这是整个 demo 最重要的设计判断：**LLM 负责「演」，代码负责「判」。**

## 4. 难度与公平性

- 真凶 `陈志远` 有 **3 条** 可被击破的谎言（不在场证明 / 去过书房 / 淋雨行踪），定罪只需 **2 条**——容错，不卡死。
- 两名无辜者各有「会让人怀疑」的秘密（苏曼偷印遗嘱、林晓的遗产动机），构成**红鲱鱼**，但都另有不在场证明，且把证据甩给他们**不会**击破任何谎言。
- 证物从一开始就全给玩家（在简报「你的证物」里）。**难点不在「找证据」，而在「想清楚该把哪件证据甩给谁」**——这才是推理。

## 5. 想加一桩新案子？

因为案子是**纯数据**，加一桩新案只需在 `src/data/cases/` 新建一个 `CaseData` 对象：写好线索、证物、三份嫌疑人 prompt + 离线剧本、`lieBreaks` 映射，然后在 `BootScene` 里换用即可。无需改任何引擎逻辑——这就是数据驱动的好处。

下一步可扩展：多案章节、对话中「解锁」隐藏证物、给每位嫌疑人加一条「情绪/信任」表，让 LLM 的反应进一步影响演出（但**仍不影响判定**）。
